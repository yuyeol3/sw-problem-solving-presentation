작업 ①

데이터 수집·정규화 파이프라인

apicollector / normalize

출처마다 형식이 제각각인 원천 데이터를, 추천 엔진이 쓸 수 있는 일관된 스키마로 정제.

  • 금감원(FSS)·온통청년 API 데이터를 규칙 기반으로 추출·정규화

  • 우대금리·가입조건이 정규화되지 않던 문제, api ↔ backend 스키마 불일치 해소
  • FSS 상품 정규화 데이터 품질 개선, 온통청년 레거시 행 제외 처리

작업 ②

LLM 데이터 보강 & 할루시네이션 방어

apicollector / llm · enrich

규칙으로 추출이 어려운 항목(우대조건·가입조건 등)은 LLM(Gemini)으로 보강. 단, 잘못된 값이 추천에 섞이지 않도록 방어.

  • 규칙 기반 추출 + LLM 보강 하이브리드 파이프라인
  • 가입조건 할루시네이션 방지 프롬프트 + 사후 검증 로직 추가
  • temperature 0.1로 낮추고 설정파일(application-dev.yml)로 분리
  • LLM 보강을 비동기로 전환, JSON 파싱 성공 기준 유연화

작업 ③

정부상품 수동입력 파이프라인

normalize / tasklet

청약·정책 상품은 공개 API가 없다. 그래서 수집 계층과 동일한 규격으로 흐르는 수동입력 경로를 새로 구축.

  • manual-products.json → 배치가 읽어 동일 파이프라인으로 정규화
  • variant_code 도입 — 청약상품 등 정부상품 속성 구분
  • API 응답에 청약상품 enum 추가로 프론트가 유형 구분 가능
  • 수동입력 상품 정규화 테스트 작성 (회귀 방지)

증빙

내 백엔드가 화면에서 이렇게 쓰입니다

Backend → UI

앞서 설명한 백엔드 작업이 실제 화면의 어느 값으로 나타나는지 — 코드가 곧 사용자 경험.

추천 리스트 적합도 점수 배지 + 기여금 환산 수익률로 정부·은행 상품을 한 기준으로 정렬
상품 상세 상세정보 API가 속성·적합도를 내려주고, 신청 URL 통일로 '신청하러 가기' 동작
금리 계산기 달성 가능 금리(기본 4.50% + 충족 우대 1.00%) — 광고 금리가 아닌 실제 받을 금리
